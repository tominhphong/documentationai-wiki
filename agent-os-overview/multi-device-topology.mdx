## Network Topology

Ba thiết bị kết nối với nhau qua ba tầng mạng: LAN nội bộ cho giao tiếp trực tiếp khi ở cùng router, Tailscale mesh overlay cho cross-network reliability, và Internet cho truy cập SaaS. Mỗi tầng có latency và use case riêng — hệ thống tự động chọn route tối ưu dựa trên context.

### LAN nội bộ (192.168.8.0/24)

Ba thiết bị và NAS Seoul đều kết nối trực tiếp với router ASUS cùng dải subnet. Đây là con đường nhanh nhất (latency thấp) khi tất cả cùng ở nhà.

- **Mac Studio:** LAN IP chưa xác nhận (xem `ifconfig en0`)
- **Mac mini M4:** `192.168.8.107`
- **UM790:** `192.168.8.174`
- **NAS Seoul:** `192.168.8.114`
- **Gateway:** `192.168.8.1` (router ASUS)

Khi Phong làm việc từ Mac Studio tại nhà và cần gọi task tới Mac mini (ví dụ: trigger GoClaw), hệ thống preferred dùng LAN IP trực tiếp để giảm latency. Tuy nhiên, nếu không xác định được LAN route (hoặc Phong đang ở coffeeshop), automatic fallback sang Tailscale.

### Tailscale mesh (100.64.0.0/10)

Overlay WireGuard encryption che phủ toàn bộ 4 node. Mỗi máy có Tailscale IP tĩnh trong CGNAT range, và MagicDNS tự động resolve hostname thành Tailscale IP — không cần hardcode IP trong script.

- **Mac Studio:** `100.98.44.123` (`phongs-mac-studio`)
- **Mac mini M4:** `100.98.146.85` (`phongrealtortxs-mac-mini`)
- **UM790:** `100.71.136.88` (`minisforum`)
- **NAS Seoul:** `100.88.222.69` (`seoul`)

Tailscale hoạt động xuyên mạng — Phong ở nhà, coffeeshop, hoặc khách sạn đều có thể reach các thiết bị qua MagicDNS hostname. Ví dụ: script trên Mac Studio chỉ cần gọi `ssh phongrealtortxs-mac-mini.tail<REDACTED>` thay vì hardcode `192.168.8.107` (địa chỉ LAN có thể thay đổi nếu DHCP renew). Tất cả lưu lượng Tailscale được encrypt — an toàn cho data nhạy cảm.

### Internet (SaaS)

Ba thiết bị truy cập Internet qua ISP để gọi tới external services:

- **Claude API** — Opus, Sonnet inference
- **Cloudflare R2** — media hosting
- **Late API (Zernio)** — social media scheduling
- **HubSpot** — CRM
- **Documentation.AI** — wiki hosting + MCP endpoint

Lưu lượng SaaS đi qua WAN từ router, không đi qua Tailscale (vì SaaS services ngoài tailnet). Phong có thể hạn chế WAN traffic bằng firewall rules trên router nếu cần, nhưng hiện tại không có giới hạn.

### Routing quy tắc

**Ưu tiên 1 — LAN nội bộ:** Nếu source và destination cùng dải `192.168.8.0/24`, dùng LAN IP trực tiếp (latency thấp, không qua Tailscale). Ví dụ: Mac mini gọi NAS Seoul qua `192.168.8.114` nhanh hơn qua Tailscale.

**Ưu tiên 2 — Tailscale MagicDNS:** Nếu source và destination không cùng LAN, hoặc LAN route không khả dụng, fallback sang Tailscale MagicDNS. Script không cần biết LAN hay Tailscale — chỉ cần resolve hostname qua DNS tự động chọn best path.

**Ưu tiên 3 — Internet (SaaS only):** External services (Claude API, R2, etc.) không có Tailscale IP — chỉ có Internet public IP. Traffic ra Internet luôn qua ISP.

### Firewall & ACL

Tailscale tailnet của Phong (user `windcvs@`) mặc định cho phép all-to-all communication — tất cả 4 node đều có thể reach nhau. Nếu Phong muốn hạn chế (ví dụ: UM790 chỉ accept SSH từ Mac Studio), có thể cấu hình ACL trong Tailscale console — nhưng hiện tại không cần. Không expose Tailscale IP ra public Internet — MagicDNS chỉ resolve trong tailnet.

## Role Allocation

Agent OS tuân theo nguyên lý "mỗi máy, một vai trò chuyên môn" — không có máy chủ trung tâm, mà mỗi thiết bị tối ưu cho loại task của nó. Mac Studio là bộ não, Mac mini là worker 24/7, UM790 xử lý Windows-only. Phong tại Mac Studio gửi directive, hệ thống dispatch task tới máy phù hợp qua Tailscale RPC.

### Task-type matrix

| Loại task | Mac Studio | Mac mini M4 | UM790 |
|-----------|-----------|-------------|-------|
| Interactive editing (code, docs, browse) | ✅ primary | ❌ | ❌ |
| Opus orchestration / planning | ✅ primary | ⚠️ fallback Sonnet | ❌ |
| Scheduled cron jobs (nightly, hourly) | ❌ | ✅ primary | ❌ |
| Long-running agents (không tương tác) | ⚠️ OK | ✅ primary | ❌ |
| GoClaw AI agent gateway | ❌ | ✅ primary | ❌ |
| Media processing (FFmpeg, ImageMagick) | ✅ primary | ⚠️ OK | ❌ |
| Windows-only apps (Hidemium, Boxphone) | ❌ | ❌ | ✅ primary |

**Ký hiệu:**
- **✅ primary** — thiết bị được ưu tiên cho task này, đã setup đầy đủ
- **⚠️ fallback / OK** — có thể chạy nhưng không ideal (latency cao, spec thấp, hoặc offline)
- **❌** — không support hoặc không nên chạy

### Routing quy tắc

**Phong ngồi tại Mac Studio → interactive task bắt đầu tại đây.** Claude Desktop ở Mac Studio trong chế độ Cowork Max 20x chạy Opus 4.7 làm orchestrator. Phong gửi yêu cầu ("write unit test cho auth.ts"), Claude quyết định local hay dispatch tới máy khác.

**Task schedule / long-running → dispatch tới Mac mini M4 via Tailscale.** Nếu task là "nightly report generation" hoặc "monitor GoClaw gateway", Mac Studio gửi job qua HTTP/SSH tới Mac mini (qua `phongrealtortxs-mac-mini.tail<REDACTED>` MagicDNS), Mac mini xử lý 24/7 không cần tương tác từ Phong.

**Windows-only task → từ Mac Studio/mini gửi job tới UM790.** Nếu cần chạy Hidemium hoặc Boxphone, Mac Studio hoặc Mac mini SSH qua Tailscale tới `minisforum` execute, kết quả (logs, screenshot) trả về qua Tailscale.

**Nếu Mac Studio offline (hiếm) → Mac mini fallback chạy Sonnet thay Opus.** Mac mini có 16GB RAM + Docker, có thể chạy Sonnet 4.6 inference local (không Opus — cần spec cao hơn). Phong không ở Mac Studio, nhưng system vẫn hoạt động — pending task queue ở Mac mini đợi Mac Studio online, hoặc execute low-priority task qua Sonnet.

### Kênh giao tiếp cross-device

**Shared state:** `~/Shared Workspace/` — truy cập bởi tất cả 3 thiết bị qua Tailscale mount (mechanism chưa chốt — xem §4 Sync Mechanism). File tracking, task status, brand/compliance guidelines đều lưu ở đây để mỗi agent thấy state mới nhất.

**RPC (gọi hàm từ xa):** HTTP hoặc SSH qua Tailscale MagicDNS hostname. Ví dụ: Mac Studio chạy `curl https://phongrealtortxs-mac-mini.tail<REDACTED>/api/schedule-job -X POST -d '{...}'` để trigger cron task ở Mac mini. Hoặc `ssh minisforum 'hidemium-cli run-profile ...'` để exec task ở UM790.

**Events (không có message bus):** Hiện tại không có centralized event queue (YAGNI — thêm khi cần). Nếu task ở Mac mini hoàn thành, nó update file ở `~/Shared Workspace/results/` — Mac Studio poll file đó hoặc dùng `fswatch` để detect change.

## Sync Mechanism

Hệ thống Agent OS lưu trữ state chia sẻ trong `~/Shared Workspace/` — file tracking, brand guidelines, compliance rules, và task progress. Cơ chế sync đảm bảo 3 thiết bị thấy state mới nhất mà không có conflict hoặc mất dữ liệu.

### Decision

Sync mechanism được chốt **2026-04-24**: **GitHub private repo (Option A)**.

- **Repository:** `phong-shared-workspace` (private) trên GitHub
- **Status:** Kiến trúc quyết định, implementation backlog

### Rationale

GitHub private repo được chọn vì:

1. **Explicit version control** — mỗi thay đổi có commit message + diff rõ ràng, thuận tiện audit trail khi cần review lịch sử.
2. **Cross-platform** — Mac + Windows + Linux đều hỗ trợ git CLI. UM790 (Windows) không bị "locked out" như iCloud.
3. **Zero infrastructure cost** — GitHub free tier đủ cho text-heavy workspace (không lưu binary media lớn).
4. **Phong đã quen flow** — account + SSH keys đã setup, 3 máy authenticated. Không cần học tool mới.
5. **Reliability + history** — Git history bảo toàn, có thể revert nếu cần. Trade-off: latency cao hơn iCloud (manual `git pull/push` thay vì real-time) — chấp nhận, có thể tự động hóa ở sprint sau.

### Architecture

Cơ chế hoạt động như sau:

**Repository structure:**
```
phong-shared-workspace/ (private GitHub repo)
├── findings.md               # Architectural decisions, device specs
├── task_plan.md             # Sprint roadmap, task list
├── progress.md              # Append-only activity log
├── brand.md                 # Voice, messaging, platforms
├── compliance.md            # TREC, Fair Housing rules
├── .gitignore               # Block: *.env, *.key, credentials.json
└── (other shared files)
```

**Workflow trên từng máy:**

1. **Session start** — Trước khi task bắt đầu: `cd ~/Shared\ Workspace && git pull` lấy state mới nhất.
2. **Session work** — Agent đọc/ghi files cần thiết (findings.md, task_plan.md, etc.). Phong làm việc bình thường.
3. **Session end** — Sau khi task xong: `git add . && git commit -m "..."` (có thể tự động qua cron + pre-commit hook). Rồi `git push origin main`.

**Conflict resolution:**

- Nếu 2 máy push cùng file, GitHub reject push thứ 2. Máy đó phải `git pull --rebase` rồi resolve conflict (thường merge tự động nếu không edit cùng line).
- Mac Studio (Lead device) là source of truth khi conflict — Phong resolve manual trên Mac Studio rồi push.
- Không cho phép force-push để tránh overwrite — mỗi change phải explicit commit + merge.

**Secret handling:**

- `.gitignore` block `*.env`, `*.key`, `credentials.json`, `secrets/` — không commit secrets vào repo.
- Sensitive files (API keys, auth tokens) lưu local trên từng máy + add vào `.gitignore` nếu cần chia sẻ (dùng 1password / bitwarden shared vault thay).

| Scenario | Likelihood | Mitigation |
|----------|-----------|------------|
| Conflict giữa 2 máy push cùng file | Low (hiếm nếu coordinate tốt) | `git pull --rebase` trên máy đó, resolve manual, push lại |

## Failure Modes & Fallback

Hệ thống Agent OS là phân tán — mỗi máy có thể fail độc lập. Section này mô tả hành vi khi thiết bị offline và fallback strategy tương ứng.

<Callout kind="warning">
Nội dung section này dựa **design-time reasoning**, chưa qua testing thực tế. Khi có incident thực, sẽ revise dựa empirical data và logs.
</Callout>

### Failure Scenarios

| Scenario | Severity | Effect | Fallback |
|----------|----------|--------|----------|
| **Mac Studio offline** | High | Không interactive work được; Opus orchestrator down | Mac mini chạy Sonnet 4.6 fallback cho urgent task; Phong dùng Claude.ai web hoặc iPhone tạm thời |
| **Mac mini M4 offline** | Medium | Scheduled cron jobs fail; GoClaw gateway xuống | Mac Studio chạy task manual; restart Mac mini qua Tailscale SSH + `sudo shutdown -r` hoặc Wake-on-LAN nếu cần |
| **UM790 offline** | Low | Hidemium / Boxphone tasks pause | Không có fallback — wait until máy lên (Windows-only, không thay thế được) |
| **Tailscale mesh down** (cả 3 máy) | High | Cross-device communication fail | LAN fallback nếu 3 máy cùng nhà (dùng LAN IP 192.168.8.x trực tiếp); nếu Phong ở remote → chờ Tailscale khôi phục (rất hiếm) |
| **Internet down** | Medium | SaaS APIs fail (Claude API, R2, Late, HubSpot) | Local tasks (code, file processing, research) vẫn chạy được; defer SaaS-dependent tasks đến khi Internet lên |
| **NAS Seoul offline** | Low | Media/backup access fail | Không block agent work; defer NAS-dependent tasks (photo batch, backup) |

### Fallback Priority Logic

Khi 1 device fail, hệ thống tuân theo quy tắc:

1. **Graceful degrade** — task critical reroute sang máy còn online nếu có. Task non-critical defer (không cancel).
   - Ví dụ: "write report" (critical) reroute Mac mini Sonnet nếu Mac Studio offline. "Archive old logs" defer.

2. **Notify Phong** qua iMessage hoặc email nếu Mac Studio offline >30 phút liên tiếp.
   - Message format: `"[Agent OS] Mac Studio offline since HH:MM. Switched to Mac mini Sonnet fallback. [Reconnect/Resume]"`
   - Giúp Phong biết system vẫn chạy hoặc có issue cần handle.

3. **Auto-retry không block session** — nếu task fail ở máy primary, không block session. Thử lại sau N giây (5–10s). Sau 3 retry fail → ghi log + notify.
   - Ví dụ: task gọi GoClaw ở Mac mini fail → log warning, thử lại sau 10s, nếu vẫn fail → defer task + notify Phong.

4. **Log mọi fallback** để Phong review sau.
   - Log entry: `[2026-04-24 10:45:30] FALLBACK Mac Studio offline; task 'generate-report' rerouted to Mac mini`
   - Log file: `~/Shared Workspace/logs/fallback.log` (push qua git sync).
