## Tổng quan mô hình

Wiki này được xây và duy trì bởi một team 4 agents chạy song song trên 4 iTerm2 windows (Win 1–4), mỗi agent là một instance `claude` CLI độc lập. Không có runtime communication giữa các agents — tất cả coordination thông qua filesystem (git + markdown files trong `plans/`).

Mô hình này là ví dụ thực tế của **multi-agent collaboration** được document trong chính wiki này.

<Callout kind="info">
Trước khi bất kỳ agent nào bắt đầu làm việc: đọc `AGENTS.md` để xác nhận role, đọc `CODEOWNERS` để xác nhận scope, đọc `plans/task-board.md` để claim task.
</Callout>

## File Ownership là bất khả xâm phạm

Mỗi module folder được assign cho một agent duy nhất. Không được edit file ngoài scope của mình — kể cả khi "chỉ sửa một chút".

Lý do:
- Parallel agents có thể cùng modify một file → merge conflicts → build fail
- Agent thiếu context về quyết định thiết kế của owner → edit không nhất quán
- Win 4 chỉ QA code của người khác, không edit — giữ nguyên tắc separation of concerns

Chi tiết đầy đủ trong `CODEOWNERS` và page [File Ownership Rules](/core-rules/file-ownership-rules).


## Task Claiming Flow

Agents không tự chọn task tùy ý. Flow chuẩn:

<Steps>
### Đọc task-board
Mở `plans/task-board.md`, tìm task trong **TODO** phù hợp với role của mình.

### Claim task
Move task sang **IN-PROGRESS** và thêm tag `[winN]`. Commit thay đổi ngay để signal cho các agents khác.

### Làm việc
Thực thi task trong worktree branch của mình. Commit thường xuyên với conventional commit messages.

### Push + chuyển sang REVIEW
Push branch, move task sang **REVIEW** trong task-board, append vào `plans/progress.md`.

### Win 4 validate
Win 4 pull branch, chạy `scripts/pre-commit-check.sh`, comment kết quả trong task-board.

### Win 1 merge
Nếu QA pass, Win 1 merge vào `main`, move task sang **DONE**, verify deploy.
</Steps>


## Cross-Scope Requests

Khi agent cần chỉnh một file ngoài scope của mình (VD: Win 2 cần thêm page vào `documentation.json`), **không được tự edit**. Thay vào đó:

1. Vào `plans/task-board.md` section `## Cross-Scope Requests`
2. Thêm entry theo format:

```
### [YYYY-MM-DD HH:MM] [winN → win1] <short title>
Reason: <why this edit is needed>
Patch:
<inline diff or clear instructions>
Status: pending
```

3. Win 1 apply patch → đổi Status thành `applied`

Ví dụ thực tế: khi Win 2 tạo page mới `core-rules/file-ownership-rules.mdx`, Win 2 viết cross-scope request để Win 1 thêm path vào `documentation.json`.


## Coordination qua Filesystem

Agents không có shared memory hay runtime IPC. Tất cả state nằm trong ba files:

| File | Mục đích | Update khi nào |
|------|----------|----------------|
| `plans/task-board.md` | Kanban trạng thái tasks | Mỗi khi claim, move, hoặc complete task |
| `plans/progress.md` | Activity log append-only | Sau mỗi commit |
| `plans/findings.md` | Gotchas, blockers, decisions | Khi phát hiện vấn đề hoặc ra quyết định quan trọng |

**Context freshness rule:** Mỗi 3–5 tasks, re-read `plans/progress.md` để sync với work của agents khác.
