## Issue 1 — Claude Code CLI 401 auth khi multiple concurrent sessions

<Callout kind="error">
**Critical:** Lỗi này block toàn bộ session của một worker agent. Phải xử lý ngay khi gặp.
</Callout>

**Symptom**

Một Claude Code CLI session mới báo lỗi authentication sau khi `/login`:

```
Error: 401 Unauthorized — authentication failed
```

Hoặc CLI không start được trong một worktree cụ thể, trong khi session khác vẫn chạy bình thường.

**Root cause**

Concurrent session conflict — khi nhiều Claude Code CLI instances chạy đồng thời với cùng một Anthropic API key, OAuth token có thể xung đột tùy theo account tier. Session mới nhất bị từ chối vì token đã được bind vào session cũ.

**Workaround**

Thay vì chạy multiple Claude Code CLI instances độc lập, dùng **Agent Team feature trong một main session**:

1. Chạy `claude` từ repo chính (không phải từng worktree riêng)
2. Trong main session, dùng Agent Team tools để spawn Win 2/3/4 as sub-agents
3. Sub-agents nhận task qua filesystem (`plans/task-board.md`, `plans/cross-scope-sprint-N.md`) thay vì separate CLI sessions

**Prevention**

Đây là kiến trúc được recommend cho multi-agent sprint. Không chạy nhiều CLI instances độc lập — chạy một main session và delegate qua Agent Team.

## Issue 2 — Haiku 4.5 input token limit nhỏ hơn Sonnet

**Symptom**

Agent được assign model `haiku` trả về lỗi:

```
Error: Prompt is too long. Max input tokens: ...
```

Prompt với context files lớn hoặc task phức tạp fail trên Haiku nhưng pass trên Sonnet.

**Root cause**

Haiku 4.5 có context window nhỏ hơn Sonnet 4.6. Long prompts trên under 3000 tokens thường an toàn, nhưng prompts phức tạp kèm nhiều file context có thể vượt giới hạn Haiku.

**Workaround**

- Strip prompt xuống essentials — bỏ instructions lặp lại, chỉ giữ task + files cần thiết
- Chuyển sang Sonnet 4.6 cho complex tasks yêu cầu large context
- Estimate safe Haiku prompt size: ~2500 tokens (bao gồm cả system prompt và file content)

**Prevention**

Dùng Haiku cho lightweight tasks (formatting, adding frontmatter, simple edits). Dùng Sonnet cho tasks cần đọc nhiều files, viết nội dung dài, hoặc phân tích phức tạp.

## Issue 3 — Bash `set -e` + `(( ))` arithmetic false trap

<Callout kind="warning">
**Lỗi này rất khó debug vì script silently exits** — không có error message, chỉ đơn giản là dừng giữa chừng.
</Callout>

**Symptom**

Bash script silently exits giữa chừng trong một vòng lặp. Không có error message, script chỉ dừng mà không có output tiếp theo. Behavior này xảy ra ngẫu nhiên tùy vào giá trị biến.

**Root cause**

`(( expr ))` trong bash trả về exit code 1 khi expression evaluates to 0 (false). Khi script có `set -e`, bất kỳ command nào trả về non-zero exit code đều trigger script exit ngay lập tức. Kết hợp hai điều này: `(( var ))` khi `var == 0` sẽ kill script.

**Bad pattern:**

```bash
set -e
count=0
for f in *.mdx; do
  (( count++ ))  # Dies when count is 0 before increment (exit code 1 from (( 0 )))
  echo "Processing $f"
done
```

**Fix:**

```bash
# Dùng [[ ]] thay (( )) cho comparison
[[ ${#var} -lt 3 ]]   # OK — returns 0 or 1 cleanly
(( ${#var} < 3 ))     # DANGEROUS với set -e nếu var length >= 3 → expr = 0 → exit 1

# Hoặc dùng += thay (( ++ ))
count=$((count + 1))  # arithmetic substitution — safe, không trigger set -e
```

**Prevention**

Trong scripts với `set -e`: luôn dùng `[[ ]]` cho conditionals. Chỉ dùng `(( ))` khi wrap trong `if (( ... )); then` — vì `if` absorbs non-zero exit codes.

## Issue 4 — Bash `set -o pipefail` + grep no-match exits script

**Symptom**

Script với `set -e` và `set -o pipefail` thoát đột ngột tại dòng có `grep` pipeline. Không có error message. Behavior chỉ xảy ra khi grep không tìm thấy match — khi có match thì bình thường.

**Root cause**

`pipefail` flag làm pipeline fail nếu bất kỳ command nào trong pipe fail. `grep` trả về exit code 1 khi không có match (expected behavior). Kết hợp với `set -e`, "no match" result từ grep kill script.

**Bad pattern:**

```bash
set -e -o pipefail
# Script dies here if "keywords:" not found
x=$(grep "keywords:" "$file" | head -1)
```

**Fix:**

```bash
# Append || true để absorb grep no-match
x=$(grep "keywords:" "$file" | head -1 || true)

# Hoặc check trước
if grep -q "keywords:" "$file"; then
  x=$(grep "keywords:" "$file" | head -1)
fi
```

**Prevention**

Khi viết scripts với `set -e -o pipefail`, luôn append `|| true` vào grep/find commands khi "no match" là valid/expected state. Đặt comment rõ lý do để tránh người sau xóa đi.

## Issue 5 — Mermaid diagrams không render trong Mintlify theme hiện tại

**Symptom**

Code block với syntax `` ```mermaid `` render as plain text — không có diagram, chỉ thấy raw Mermaid code.

**Root cause**

Mintlify theme hiện tại của Documentation.AI chưa enable Mermaid plugin. Code block được treated như normal code block và hiển thị as-is.

**Workaround**

Ba lựa chọn thay thế, theo thứ tự prefer:

| Lựa chọn | Khi nào dùng | Ví dụ |
|----------|--------------|-------|
| ASCII art | Simple flows, trees | `A → B → C` |
| Markdown table | Data relationships | Header/row structure |
| Mintlify Steps component | Sequential processes | `<Steps>` / `<Step>` |

**Ví dụ ASCII thay thế Mermaid:**

```
Claude Desktop
├── Desktop Commander MCP
│   └── Terminal (iTerm2/it2)
├── Zernio MCP
│   └── Social Media APIs
└── Remotion
    └── Video render pipeline
```

**Future fix**

Check [Mintlify docs](https://mintlify.com) cho Mermaid plugin support. Khi plugin available, cần Win 1 enable trong `docs-config` và convert ASCII art → Mermaid diagrams.

## Issue 6 — Plans/cross-scope merge conflicts khi 2+ workers append parallel

**Symptom**

`git pull` hoặc `git merge` báo conflict trong `plans/cross-scope-sprint-N.md`:

```
<<<<<<< HEAD
### [2026-04-20 14:28] [win4 → win1] Add page-A to nav
=======
### [2026-04-20 14:29] [win2 → win1] Add page-B to nav
>>>>>>> origin/win3-workflow
```

**Root cause**

Hai worker agents cùng append vào `## Pending` section trong cùng khoảng thời gian ngắn. Git không biết cách merge vì cả hai đều append vào cuối file — không có context line chung.

**Fix — giữ CẢ HAI entries:**

```
### [2026-04-20 14:28] [win4 → win1] Add page-A to nav
Patch: ...
Status: pending

### [2026-04-20 14:29] [win2 → win1] Add page-B to nav
Patch: ...
Status: pending
```

Xóa conflict markers (`<<<<<<<`, `=======`, `>>>>>>>`), giữ nguyên nội dung cả hai. Brain (Win 1) resolve thủ công — áp dụng cả 2 patches rồi move to `## Applied`.

**Prevention — Sprint 5+**

Có thể split cross-scope file per-worker: `cross-scope-sprint-5-win2.md`, `cross-scope-sprint-5-win3.md`, v.v. Brain merge tất cả khi close sprint. Cách này eliminate conflict hoàn toàn.

```bash
# Sau khi resolve conflict thủ công
git add plans/cross-scope-sprint-4.md
git commit -m "fix: resolve cross-scope merge conflict — keep both entries"
```

## Issue 7 — GitHub OAuth thiếu workflow scope khi push

<Callout kind="warning">
**One-time fix per machine.** Sau khi refresh token với workflow scope, lỗi này không xuất hiện lại trừ khi token bị revoke.
</Callout>

**Symptom**

`git push` bị reject với message:

```
refusing to allow an OAuth App to create or update workflow
  '.github/workflows/validate-mdx.yml'
  without 'workflow' scope
```

**Root cause**

GitHub OAuth token hiện tại được tạo trước khi repo có `.github/workflows/` directory, hoặc token được tạo mà không include `workflow` scope. Lỗi này xuất hiện khi bạn modify bất kỳ file nào trong `.github/workflows/`.

**Fix — one-time action per machine:**

```bash
# Refresh token và add workflow scope
gh auth refresh -s workflow --hostname github.com
```

Browser sẽ mở hoặc hiển thị device code:

```
! First copy your one-time code: XXXX-XXXX
Press Enter to open github.com in your browser...
```

1. Copy code → nhấn Enter
2. Dán code vào GitHub device authorization page
3. Click "Authorize GitHub CLI"
4. Verify sau khi complete:

```bash
gh auth status
# Token scopes phải bao gồm: ..., workflow
```

**Prevention**

Khi setup agent trên máy mới, luôn chạy `gh auth refresh -s workflow` ngay từ đầu trước khi bắt đầu sprint. Ghi vào setup checklist.

## Issue 8 — Mintlify MDX v3 không parse được `&#123;#anchor&#125;` heading syntax

<Callout kind="error">
**Blocking deploy.** Documentation.AI reject entire file với "Could not parse expression with acorn" error. Phát hiện Sprint 4.
</Callout>

**Symptom**

Deploy logs báo:

```
Invalid content format in 'agent-os-overview/known-issues.mdx':
Failed to parse MDX content: Could not parse expression with acorn (line 20, column 73)
```

Line 20 chứa heading kiểu GFM anchor:

```markdown
## Issue 1 — Some title {#issue-1}
```

**Root cause**

MDX v3 parser (acorn) đọc `{...}` ở bất kỳ đâu ngoài code fence như JSX expression. `#issue-1` bắt đầu bằng `#` không phải valid JavaScript identifier → acorn fail.

GFM (GitHub Flavored Markdown) support `{#anchor}` heading IDs, nhưng Mintlify dùng pure MDX — không cho pass-through markdown extension.

**Fix**

Remove all manual `{#anchor}` syntax — Mintlify tự sinh heading IDs từ text.

```bash
# Strip tất cả {#...} từ H2 headings
sed -i '' 's/ {#[a-z0-9-]*}$//' path/to/file.mdx
```

Update TOC: thay linked list `[text](#anchor)` → numbered list plain text. On-page TOC sidebar (Mintlify auto-generate) vẫn work.

**Prevention**

- **Rule:** Không dùng `{...}` trong markdown content trừ khi thực sự là JSX expression hợp lệ
- Pre-commit `scripts/pre-commit-check.sh` nên thêm check regex: `\{#[^}]*\}` ngoài code fence → warning
- Future: codify vào CLAUDE.md MDX conventions section
