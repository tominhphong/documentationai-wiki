# Case Study: Mortgage Compare App from Idea to Deploy in One Week

## Context

### Business Need

Vietnamese-American homebuyers often bring complex financial profiles to consultations — 1099 contractor income, multiple co-borrowers, gift funds, varying down payment scenarios. Phong needed a tool to run side-by-side mortgage comparisons live during client meetings, not after.

### Starting Point

No existing app. Phong had a clear mental model of what a "financial advisor" would check before advising a buyer, and a list of 12+ criteria he wanted the tool to cover.

### Goal / Success Criteria

- Compare up to 4 mortgage scenarios side-by-side in real time
- Cover 12+ financial advisor criteria: TCO, tax savings (2026 rules), rent vs buy, refinance break-even, recast analysis, opportunity cost
- Deploy on free tier — $0/month operational cost
- Accessible from any device (responsive design)
- Data persists across sessions (cloud sync)

- Dark mode (next-themes) — fully implemented in one sprint
- DTI (debt-to-income) affordability calculator

- Scenario comparison grid (up to 4 scenarios)

**Day 5–6 (Sprint 3): Financial advisor criteria**

- TCO (total cost of ownership) over 5/10/30 year horizons
- Tax 2026 rules: Mortgage Interest Deduction capped at $750K, SALT deduction capped at $10K
- MCC (Mortgage Credit Certificate) integration for first-time buyers
- Rent vs buy break-even analysis
- Refinance break-even (months to recoup closing costs)
- Recast calculator (lump-sum principal + recast fee)
- Opportunity cost: what the down payment could earn if invested instead

**Day 7 (Sprint 4): Cloudflare backend**

- Cloudflare D1 (SQLite at the edge) — stores saved scenarios
- Cloudflare R2 — stores exported PDF comparisons
- Cloudflare Worker API — REST endpoints for save/load/export
- Frontend cloud sync: scenarios persist across devices
- Deploy: Vercel (frontend, free tier) + Cloudflare (backend, free tier)

## Technical Approach

### Stack

| Layer | Technology | Why |
|-------|-----------|-----|
| Frontend | Next.js 16.2.4 + React 19 | App Router, server components, edge-ready |
| Styling | Tailwind v4 + shadcn/ui | Fast theming, accessible components |
| Dark mode | next-themes | Zero-config theme switching |
| Backend | Cloudflare Worker (TypeScript) | Edge latency, free tier generous |
| Database | Cloudflare D1 (SQLite) | Relational, free tier, co-located with Worker |
| Storage | Cloudflare R2 | S3-compatible, free egress, PDF exports |
| Deploy | Vercel + Cloudflare | Both free tier, $0/month total |

### Multi-Agent Pattern

```
Brain (Opus)
  ├── Worktree agent-ui      → src/components/**, src/app/**
  ├── Worktree agent-math    → src/lib/calculators/**
  ├── Worktree agent-state   → src/lib/store/**, src/hooks/**
  └── Worktree agent-worker  → cloudflare/worker/**, cloudflare/d1/**
```

Sprint 4 added a 5th agent:

```
  └── Worktree agent-sync    → src/lib/sync/**, src/app/api/**
```

### Key Design Decisions

1. **Edge-first backend:** Cloudflare Workers run at the network edge — sub-50ms response times globally, included in the free tier up to 100K requests/day.
2. **D1 over Postgres:** For a single-user tool, D1 is simpler and free. No connection pooling, no cold starts.
3. **shadcn/ui over custom components:** Accessible by default, Tailwind-native, copy-paste installation means no dependency lock-in.
4. **Calculation accuracy over UI polish:** Sprint 1 prioritized correct math. UI refinement came after all calculations were verified against known-good mortgage amortization tables.

## Workflow in Action

### Spawning the Sprint 4 Backend Agents

```
# Brain prompt for Sprint 4
"Sprint 4: Cloudflare backend integration.

Spawn 2 agents in parallel:

agent-worker (worktree: worktree-agent-worker):
  File ownership: cloudflare/**
  Task: Create Worker with D1 database schema + R2 bucket integration.
  Endpoints needed:
    POST /scenarios — save scenario JSON to D1
    GET  /scenarios/:id — load saved scenario
    POST /export — store PDF to R2, return presigned URL
  Use TypeScript. Handle CORS for Vercel frontend origin.

agent-sync (worktree: worktree-agent-sync):
  File ownership: src/lib/sync/**, src/hooks/useCloudSync.ts
  Task: Frontend sync hook. On scenario save, call POST /scenarios.
  On app load, offer to restore last saved session.
  Use SWR for cache + revalidation.

Both agents report back with: file count, line count, any open questions.
Do NOT touch each other's files."
```

### The Contract Mismatch Bug (and 3-Minute Fix)

After Sprint 4, a smoke test revealed that `agent-worker` and `agent-sync` had a contract mismatch:

- `agent-worker` expected `POST /scenarios` body: `{ inputs_json: string }`
- `agent-sync` sent: `{ inputs: object }` (non-serialized)

**How it was caught:** Brain ran a smoke test prompt against the live Worker endpoint after deploy, observed a 400 error, and read both agents' code side by side.

**Fix (3 minutes):**

```
# Brain diagnosis prompt
"agent-worker expects inputs_json (string). agent-sync sends inputs (object).
Fix: update useCloudSync.ts to JSON.stringify before POST.
No other file changes needed."
```

One targeted edit to `src/hooks/useCloudSync.ts`. No rewrite, no rollback. This is the value of clear file ownership — the bug was isolated to one file.

### Example: Rent vs Buy Calculation

```typescript
// src/lib/calculators/rent-vs-buy.ts
export function calcRentVsBuyBreakeven(params: RentVsBuyParams): number {
  const monthlyOwningCost = params.mortgage + params.tax + params.insurance
    + params.hoa + params.maintenance - params.taxSavingsMonthly;
  const monthlyRentCost = params.monthlyRent * (1 + params.rentInflationRate);
  const monthlySavings = monthlyRentCost - monthlyOwningCost;
  if (monthlySavings <= 0) return Infinity; // never breaks even
  return Math.ceil(params.closingCosts / monthlySavings);
}
```

## Challenges and Lessons

### Challenge 1: Tailwind v4 breaking changes

Tailwind v4 changed how config is loaded — no more `tailwind.config.js` by default. The initial scaffold used v3 patterns that failed silently.

**Fix:** Agent-ui switched to the v4 CSS-native config (`@import "tailwindcss"` in globals.css). Lesson: always check the major version of every dependency before scaffolding.

### Challenge 2: D1 SQLite dialect differences

D1 uses SQLite, but some SQL patterns common in Postgres (e.g., `RETURNING *` in some contexts, certain JSON operators) behaved differently.

**Fix:** agent-worker ran all schema migrations against a local D1 emulator (`wrangler d1 execute --local`) before deploying to production. Added this step to the sprint checklist.

### Challenge 3: Vercel + Cloudflare CORS

The Cloudflare Worker initially rejected requests from the Vercel preview URL because CORS headers only allowed the production domain.

**Fix:** Worker reads `ALLOWED_ORIGINS` from environment variable — supports comma-separated list including Vercel preview URLs (`*.vercel.app`).

## Replication Guide

<Steps>
  <Step title="Scaffold Next.js with Tailwind v4 + shadcn">
    - Run: `npx create-next-app@latest --typescript`
    - Install Tailwind v4 per official docs (v4 has different config than v3)
    - Add shadcn/ui components with `npx shadcn-ui@latest add button input`
    - Verify Tailwind renders (e.g., test color classes) before adding business logic
  </Step>
  <Step title="Build calculator logic first, UI second">
    - Write pure TypeScript functions in `src/lib/calculators/` (no React dependencies)
    - Test each function with known inputs: `calcMortgagePayment({principal: 300000, ...}) → 1524.13`
    - Verify math against external mortgage amortization tables or financial calculators
    - Wire up UI only after all calculations pass tests
  </Step>
  <Step title="Set up Cloudflare Worker + D1">
    - Run: `npm create cloudflare@latest` to scaffold Worker project
    - Define D1 schema in `src/db/schema.sql` (tables, columns, types)
    - Create migration file: `src/db/migrations/001_initial.sql`
    - Test locally: `wrangler d1 execute local-db --file=src/db/migrations/001_initial.sql`
  </Step>
  <Step title="Define the API contract before splitting agents">
    - Write a contract doc (TypeScript types or OpenAPI) specifying request/response shapes
    - Example: `POST /scenarios {inputs_json: string} → {id: string, created_at: ISO8601}`
    - Share contract with both frontend and backend agents before they start coding
    - This prevents integration bugs like inputs vs inputs_json mismatches
  </Step>
  <Step title="Run a smoke test before calling the sprint done">
    - Brain spawns a test sequence: create scenario via UI → verify POST /scenarios succeeds → reload page → verify scenario reloads from cloud
    - Check Network tab: confirm API calls return 200, not 400 or 500
    - Verify data matches: input values in form → calculated results in table
    - Only mark sprint done after smoke test passes end-to-end
  </Step>
</Steps>
